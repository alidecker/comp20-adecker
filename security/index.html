Introduction

This paper presents the security weaknesses in a map website. The product analyzed is a web application that gets the user's current location, sends it to a database for storage and processing, and then displays the locations of other users and nearby landmarks on a map. This web application also calculates the nearest landmark to the user and, upon clicking on the user's icon, displays a line between the user's icon and this landmark. The database used is MongoDB, and the map is made using the Google Maps API.

Methodology

First, the client side sites (Landmark Locator and https://whispering-hollows-86285.herokuapp.com/checkins.json) were examined and toyed with in Google Chrome. Second, the client-side code (index.html) was examined and analyzed. Lastly, the server code was examined (index.js) and toyed with using curl on the command line. Throughout this process, online articles were consulted to gain exposure and understanding about web security issues (see References section).

Abstract of Findings

Issues Found

  1. 1. Insert Inappropriate Query String in /checkins.json

    On the checkins page, the user can type a query string requesting to see any checkin NOT equal to a particular username. This could result in a data breach, since the user could see the checkins of users - their usernames, locations, and times visited - that they are not authorized to see. Also, as presented in the video below, a nefarious checkin (e.g., a login that is actually JavaScript - see 2. Database Injection) could redirect the user to an unwanted place or gather their information and store it in an unwanted place.

    This issue is high severity for two reasons:

    • The user can see other users' data with merely the knowledge of their own username. Also, on many sites, there is only one instance of a given username; thus, if a user searches for anyone but himself, he gets everyone else's checkins because he is the only one with his username.
    • If a previous user did some cross-site scripting and had a checkin value that is actually JavaScript code, this could redirect the current user to a bad site and give them a virus, or somehow extract data from them and save it somewhere.

    The solution is not to allow the user to type '[$ne]' into the query string. Since this web application uses MongoDB, it makes sense to use the module "mongo-sanitize", as described in the screenshot below.

  2. 2. Database Injection: "login" Value in /sendLocation

    Since index.js does not check that the login is legitimate (e.g., that it contains only alphanumeric characters, or that it is below a certain character length), a user could send JavaScript code as the login. This would then be executed by the page whenever that login should be displayed on the page. In this particular web application, the login executes as JavaScript in /checkins, but not in that checkin's info window in index.html. It likely does not read the login as code in the map because the Google Maps API has safeguards against this for their info windows.

    To demonstrate this, I added a checkin to MongoDB using curl in the command window. The login field is JavaScript code that redirects to the MySpace homepage.

    When you click on this new entry on the map screen (index.html), the info window shows no login, and it does not redirect to MySpace. It likely does not read the login as code in the map because the Google Maps API has safeguards against this for their info windows.

    When you visit the /checkins page, however, the site does read the new login as JavaScript, execute it, and thus redirect you to MySpace.

    This issue is high severity because it would allow a user to redirect other users to a nefarious site, thus giving them viruses or something similarly bad. Besides redirection, other code could be injected to do other bad things, such as collect user information and send it to an undesirable place.

    The solution is to replace special characters with their HTML encoded counterparts (e.g., '<' will become '&lt'). This way, what the user originally typed will be displayed, but it will not be executed as code.

  3. 3. Non-Legitimate Latitude & Longitude

    POST /sendLocation does not check that the user entered a legitimate latitude nor longitude. However, the Google Maps API will display only legitimate coordinates; thus, if the coordinates are not legitimate, the user's position will not appear on the map and an error will ensue.
